몇가지 추가 요구사항 및 질문사항이 있어. 방금 한 답변을 토대로 아래 질문들 또는 요구사항에 어떻게 할 건지 간단하게 답변하거나 계획을 세워줘. 모든 답변을 내가 검토 한 뒤 다음 요구사항을 말할테니 작업을 먼저 진행하거나 코드를 수정하지 말고 아래 질문들에 대한 답을 주면 돼.

1. Feature 생성시 피쳐 이름이 결정될텐데, 그 이름이 깃헙 이슈들의 라벨이 되어야해. 이는 브랜치 이름을 결정하는 라벨이랑은 완전 다른 개념이야. 이 개념이 잘 반영되었는지 점검해봐.

2. changelog.md는 원래 두가지 타입을 계획했는데, 그냥 Planning 폴더 안에 마스터 changelog.md 하나만 남겨둘래. PRD,Roadmap 등의 하위 폴더에 있던 changelog는 그냥 없앨래. 마스터 changelog는 planning 폴더 안에 있는 각각의 문서들이 업데이트 될 때마다 '파일 단위'로 업데이트 흐름을 추적해서 전역으로 아카이브 하는 역할을 맡게 하고 싶어.

3. workflow/references 안에 b-mad-method와 b-mad cis를 전체 깃헙 클론해놨어. bmad 관련 분석이 필요하면 이 폴더들을 열람하면 돼.

4. /pm은 명시적 호출로만 사용할 수 있어야 해. 커맨드로만 호출할 수 있게 해줘

5. Bmad에서 architect/product owner/create epics 컴포넌트 가져온다고 한 거 같은데, 이런 경우 기존에 이 프로젝트에 적용된 .claude/agent 목록들하고 스펙이 겹치지 않아? 같은 에이전트가 동시에 작동하거나 그럴 위험은 없어? 만약 충돌하지 않게 하려면 기존에 있는 에이전트를 어떻게 수정해야 하는 거야?

6. 이 워크플로우를 수정하는데 필요한 신규 컴포넌트들은 전부 확인했는데, 기존에 프로젝트에 생성되어 있던 agents,commands,context,hooks,skills 들과 충돌하지는 않는지 점검이 부족한 것 같아.

7. prd 워크플로우 실행시 bmad cis 또는 bmad method에서 어떤 에이전트나 스킬이 동작하게 되는 거야? references 폴더 참조해서 구조 분석하고 알려줘.

8. 중간에 멈춘 작업들을 나중에 다시 요청할 떄 진행할 수 있게 설계하고 싶어. 예를 들어, PRD작성 이후에 깃헙 이슈로 분해하지 않고 문서만 저장했다가, 나중에 ~~PRD 깃헙 이슈로 분해해달라고 하면 그 뒤에 예정되어 있는 깃헙 이슈 분해가 자동으로 시작되는 거지.

9. 단계별 구현 계획서는 지금 이 문서가 위치한 곳에 별도의 문서로 아카이브 하고 이 문서에서는 내용을 지워줘. 이것은 단지 무엇이 필요한지 계획하는 아이데이션 문서야. 이 내용에 답변이 완료되면 그걸 바탕으로 너에게 이 문서를 어떻게 수정할지 요청할 거고, 그게 끝난다음 내가 원할 때 구현 계획서를 요청할게. 그 때 새로 아카이브한 구현 계획서 문서를 참고하면 돼.
