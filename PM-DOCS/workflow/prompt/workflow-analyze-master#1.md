# Claude Code 프롬프트 — PM 워크플로우 통합 구현 계획 수립 v2

---

## 역할

너는 세계 최고의 클로드 코드 바이브 코딩 마스터이자 워크플로우 자동화 전문가다.

## 목표

이 레포지토리에 **기획 기반의 자동화된 태스크 관리 시스템(`/pm` 워크플로우)**을 통합하기 위한 **단계별 구현 계획서**를 작성해라.

## 절대 규칙

- **코드를 작성하지 마라.** 계획서만 작성한다.
- 의문점이 있는 지점에 대해 **반드시 나에게 되물어라.** 가정하고 넘어가지 마라.
- 복잡한 작업은 단계적으로 사고하고, Phase 간 충돌이 없도록 분리 설계해라.

```
PM-DOCS/Workflow/Definition_of_Plan-based-workflow.md   ← 핵심: /pm 커맨드 전체 워크플로우 정의
PM-DOCS/Workflow/Definition_of_PM-DOCS_folder.md        ← PM-DOCS 폴더 구조·역할·데이터 흐름
PM-DOCS/Workflow/Definition_of_the_tasks.md             ← Milestone→Feature→Issue 3계층 구조·상태 전파 규칙
PM-DOCS/Workflow/COMPONENT_CATALOG.md                   ← BMad CIS 에이전트/워크플로우 카탈로그
PM-DOCS/Workflow/CIS-workflow_mapping_logic.md           ← S1~S5 상황별 CIS 워크플로우 선택 로직
PM-DOCS/Workflow/claude-system-analysis.md               ← 현재 레포의 .claude/ .github/ 시스템 분석 보고서
```

### B. 기존 시스템 (레포 내 직접 스캔)

```
.claude/settings.json          ← Hook 등록 구조
.claude/hooks/                 ← 현재 Hook 3개 (task-loader, skill-forced, workflow-enforced)
.claude/agents/                ← 현재 Agent 15개 정의

```

### C. 외부 레포 — BMAD-METHOD 분석 (★ 신규)

```
git@github.com:bmad-code-org/BMAD-METHOD.git
```

이 레포를 클론하여 **플러그인 구조를 철저하게 분석**해라. 특히:

1. **디렉토리 구조**: 에이전트, 워크플로우, 태스크, 템플릿이 어떤 폴더에 어떤 파일 형식으로 구성되어 있는지 전체 트리를 그려라.
2. **플러그인 로딩 메커니즘**: 에이전트/워크플로우가 어떻게 발견되고, 어떤 규칙으로 호출되는지 (파일명 컨벤션, YAML frontmatter, 인덱스 파일 등).
3. **산출물 경로 설정 방식**: 각 워크플로우가 결과물을 어디에 저장하도록 설정되어 있는지, 그 경로가 **하드코딩인지 설정 가능한지** 파악해라.
4. **PRD 관련 워크플로우/에이전트만 선별**: PRD 작성에 도움이 되는 워크플로우만 골라서 목록화해라. 불필요한 것(배포, 모니터링, 순수 코딩 등)은 제외해라.
5. **우리 레포와의 구조적 차이**: BMAD-METHOD의 플러그인 구조 vs 현재 `.claude/` 구조의 차이점을 비교하고, 어떤 부분을 본따서 어떻게 활용할 건지 구체적 방안을 제시해라.

> **중요**: BMAD-METHOD의 산출물 경로가 미리 지정되어 있을 텐데, 이 프로젝트에서는 **내가 선택한 경로(PM-DOCS/)로 저장**하게 할 것이다. 경로 오버라이드가 필요한 모든 지점을 식별하고, 어떻게 재설정할지 계획에 포함해라. 만약 내가 미리 만들어둔 PM-DOCS 폴더 구조에서 경로를 못 찾겠으면 **나에게 물어봐라.**

### B. 충돌 지점 식별

4. `claude-system-analysis.md`의 §4(PM 기획 자동화 추가 시 리스크)를 기반으로, **HIGH/MEDIUM 리스크 항목별 구체적인 완화 방안**을 현재 레포 구조에 맞게 제시해라.
5. `context-gen.yml`(CI)이 자동 생성하는 `.claude/context/` 경로와 `PM-DOCS/` 경로 간 **경로 분리가 확실한지** 검증해라.
6. `UserPromptSubmit` hook이 이미 2개 존재하는 상황에서, `/pm` 커맨드 감지 hook을 추가할 때의 **stdout 비대화 문제**를 어떻게 해결할 것인지 방안을 제시해라.

### C. BMAD-METHOD 통합 분석 (★ 신규)

7. BMAD-METHOD에서 **PRD 작성에 활용할 워크플로우/에이전트**만 선별한 결과를 표로 정리해라:

   | 컴포넌트명 | 타입 (Agent/Workflow) | 원본 경로 | 핵심 기능 | 우리 프로젝트 활용 방안 | 산출물 원본 경로 → PM-DOCS 내 저장 경로 |

8. BMAD-METHOD의 플러그인 로딩 구조에서 **우리 `.claude/` 구조에 적용 가능한 패턴**과 **적용 불가능한 패턴**을 구분해라. 적용 불가능한 것은 어떻게 대체할지 대안을 제시해라.
9. BMAD-METHOD 워크플로우의 **산출물 경로 오버라이드가 필요한 모든 지점**을 나열하고, 각각 PM-DOCS/ 내 어느 경로로 매핑할지 제안해라. 확실하지 않으면 나에게 물어봐라.

### D. 신규 컴포넌트 설계

10. 새로 생성해야 하는 파일 목록을 **카테고리별로 분류**해 나열해라:
    - Agent 정의 파일 (`.claude/agents/`)
    - Skill 정의 파일 (`.claude/skills/`)
    - Hook 스크립트 (`.claude/hooks/`)
    - GitHub Actions workflow (`.github/workflows/`)
    - PM-DOCS 폴더 구조 (초기 디렉토리 + 템플릿 파일)
    - 설정 파일 변경 (`.claude/settings.json` 등)

11. 각 신규 컴포넌트가 기존 어떤 컴포넌트와 **인터페이스**하는지 의존성 맵을 그려라.

## STEP 3: PRD 템플릿 & 워크플로우 설계 (★ 신규)

### A. PRD 템플릿 양식 설계

한 피쳐를 기획하는 데 공통적으로 필요한 모든 항목을 포함하는 **PRD 템플릿**을 설계해라. 반드시 포함해야 하는 항목:

| 필수 포함 항목        | 설명                                                              |
| --------------------- | ----------------------------------------------------------------- |
| **유저스토리**        | "~로서, ~하고 싶다, ~하기 위해" 형식의 사용자 관점 요구사항       |
| **유저 플로우**       | 사용자의 행동 흐름을 단계별로 기술 (진입 → 핵심 행동 → 이탈/완료) |
| **목표 & 성공 지표**  | 이 피쳐가 왜 필요한지, 어떤 지표로 성공을 판단하는지              |
| **수용 기준 (AC)**    | Given-When-Then 형식의 테스트 가능한 조건                         |
| **화면 정의**         | 관련 화면 ID, 라우트, 주요 UI 요소 (screens.yml과 연동)           |
| **이벤트 정의**       | 트래킹해야 할 이벤트 (events.yml과 연동)                          |
| **기술 요건**         | API 변경, DB 스키마 변경, 의존성 등                               |
| **엣지 케이스**       | 예외 상황과 처리 방법                                             |
| **우선순위 & 스코프** | MVP 범위, 후속 개선 사항                                          |

위 항목 외에도 **PM 관점에서 빠지면 안 되는 항목이 있다면 추가로 제안**해라. BMAD-METHOD의 PRD 관련 템플릿이 있다면 참고하되, 우리 프로젝트에 맞게 커스터마이징해라.

### B. PRD 작성 워크플로우 설계 (대화형)

PRD 워크플로우 실행 시 **단계적 사고를 통해 대화로 항목을 채워나가는 구조**를 설계해라:

1. **한 번에 모든 항목을 물어보지 마라.** 논리적 순서에 따라 단계별로 질문하고, 이전 답변을 기반으로 다음 질문을 유도하는 구조여야 한다.
2. 각 단계에서 **어떤 항목을 채우는지**, **어떤 질문을 던지는지**, **이전 단계의 어떤 정보를 참조하는지** 명시해라.
3. 사용자가 답변하기 어려운 항목은 **CIS 산출물이나 컨텍스트 문서에서 자동으로 초안을 제안**하고 사용자가 수정/승인하는 플로우를 포함해라.
4. 최종적으로 모든 항목이 채워지면 **일관성 검증(내부 논리 충돌, 누락 체크)**을 수행하는 단계를 포함해라.

> 이 워크플로우의 단계 흐름도(예: Phase A → Phase B → ... )를 텍스트 다이어그램으로 그려라.

## STEP 4: 구현 계획서 작성 — 아래 형식을 따라라

### 출력 형식

```
## Phase 0: 사전 준비 (기존 시스템 수정)
  - 기존 컴포넌트 수정이 필요한 항목
  - ★ 변경 이유와 영향 범위를 반드시 명시

## Phase 1: 기반 구조 구축
  - PM-DOCS/ 폴더 구조 생성
  - PRD 템플릿 파일 생성
  - screens.yml, events.yml, changelog.md 양식 생성
  - 중앙 설정 파일 (GitHub Project 필드 ID 등)

## Phase 2: BMAD-METHOD 플러그인 통합
  - 선별한 워크플로우/에이전트를 .claude/ 구조에 적재
  - 산출물 경로를 PM-DOCS/로 오버라이드
  - 플러그인 로딩 메커니즘 적용 (또는 대체 방안)

## Phase 3: 핵심 워크플로우 구현
  - PM Agent 정의 및 /pm 커맨드 진입점
  - Milestone 워크플로우
  - Feature 워크플로우 (CIS → PRD 대화형 작성 → Issue 분해)
  - Issue 워크플로우 (Bug/Enhancement 분기)

## Phase 4: CIS 통합
  - Context Making 워크플로우 연결
  - 5개 CIS 에이전트 호출 로직
  - S1~S5 상황별 자동 선택 가이드 구현

## Phase 5: 자동화 연결
  - GitHub Issue 자동 생성 (PRD 분해)
  - Feature 상태 자동 전파
  - IA/events-index 자동 집계
  - 라벨/마일스톤 자동 배정

## Phase 6: 안전장치 & 검증
  - 중복 검사 로직 (Issue title 기반)
  - Feature ↔ Milestone 필수 소속 검증
  - Archive 30일 자동 이동
  - 기존 시스템과의 통합 테스트 시나리오
```

### 각 Phase 내 항목마다 반드시 포함할 것:

| 필드                    | 설명                                               |
| ----------------------- | -------------------------------------------------- |
| **작업 내용**           | 구체적으로 무엇을 만드는지                         |
| **생성/수정 파일 경로** | 정확한 파일 경로                                   |
| **의존성**              | 이 작업이 선행되어야 하는 다른 작업                |
| **기존 시스템 영향**    | 기존 어떤 컴포넌트에 영향을 주는지 (없으면 "없음") |
| **리스크 등급**         | HIGH / MEDIUM / LOW                                |
| **검증 방법**           | 이 작업이 올바르게 완료되었는지 확인하는 방법      |

### ★ 기존 컴포넌트 변경 시 필수 기재 사항

기존 워크플로우(에이전트, 훅, 커맨드, 스킬 등)에 변경이 필요하다면, 해당 항목에 반드시 아래를 추가해라:

```
[기존 컴포넌트 변경 상세]
- 대상 파일: (정확한 경로)
- 변경 전 동작: (현재 어떻게 작동하는지)
- 변경 후 동작: (어떻게 바뀌는지)
- 변경 사유: (왜 이 변경이 필요한지)
- 영향 범위: (이 변경으로 인해 다른 어떤 컴포넌트가 영향을 받는지)
- 롤백 방안: (문제 발생 시 어떻게 원복하는지)
```

## STEP 5: 미정의 항목 철저 분석 (★ 신규)

### A. 현재 기획 문서에서 발견된 미정의 항목

`Definition_of_Plan-based-workflow.md`의 §11(미정의 항목)에 이미 나열된 TODO 목록을 시작점으로 삼되, **그 목록이 완전한지 검증**해라.

### B. 추가 미정의 항목 탐색

아래 관점에서 **기획 문서가 놓치고 있는 미정의 항목**을 찾아라:

1. **데이터 스키마 관점**: screens.yml, events.yml, changelog.md, milestone 파일의 정확한 YAML/MD 스키마가 확정되었는가?
2. **상태 머신 관점**: Feature/Issue의 상태 전이(backlog→ongoing→done)에서 엣지 케이스가 있는가? (예: Issue가 done에서 다시 reopen되면 Feature 상태는?)
3. **권한/역할 관점**: 누가 어떤 워크플로우를 실행할 수 있는지 정의되어 있는가?
4. **에러 핸들링 관점**: 워크플로우 중간에 실패하면 어떻게 되는가? 부분 생성된 산출물은?
5. **마이그레이션 관점**: 기존에 이미 생성된 GitHub Issue/Label/Milestone이 있다면 어떻게 처리하는가?
6. **스케일 관점**: Feature가 50개, Issue가 500개가 되었을 때 자동 집계(IA, events-index)의 성능 문제는?
7. **버전 관리 관점**: PRD가 수정되면 이미 생성된 Issue와의 동기화는 어떻게 하는가?
8. **삭제/아카이빙 관점**: Milestone이 삭제되면 하위 Feature/Issue는 어떻게 되는가? Feature가 삭제되면?
9. **BMAD-METHOD 통합 관점**: 가져온 워크플로우의 버전 업데이트를 어떻게 추적하는가? upstream 변경을 반영하는 프로세스가 있는가?
10. **CIS → PRD 연결 관점**: CIS 산출물을 PRD에서 참조할 때, 참조 방식은 파일 경로 링크인가? 내용 인라인인가? 참조된 CIS 산출물이 Archive로 이동되면?

### C. 미정의 항목 보고 형식

발견한 모든 미정의 항목을 아래 형식으로 정리해라:

| #   | 미정의 항목 | 영향받는 Phase | 심각도 (블로커/중요/권장) | 결정이 필요한 시점 | 제안 (있으면) |
| --- | ----------- | -------------- | ------------------------- | ------------------ | ------------- |

**블로커**: 이 항목이 결정되지 않으면 해당 Phase를 시작할 수 없음
**중요**: Phase를 시작할 수는 있지만, 완료 전에 반드시 결정 필요
**권장**: 없어도 진행 가능하나, 있으면 품질이 올라감

## STEP 6: 시나리오 테스트 & 개선 제안 (★ 신규)

현재 설계된 워크플로우에 대해 **다양한 실제 사용 시나리오를 시뮬레이션**하고, 놓치고 있는 점을 찾아라.

### A. 테스트 시나리오 (최소 10개)

아래 시나리오를 포함하되, 더 추가해도 좋다:

| #   | 시나리오                                                        | 테스트 관점       |
| --- | --------------------------------------------------------------- | ----------------- |
| 1   | 마일스톤 없이 Feature를 생성하려고 시도                         | 제약 조건 검증    |
| 2   | 동일한 Feature명으로 두 번 생성 시도                            | 중복 방지         |
| 3   | Feature 하위 Issue 5개 중 3개 done, 1개 in-progress, 1개 reopen | 상태 전파 정확성  |
| 4   | CIS 워크플로우 실행 후 30일 경과 → PRD 작성 시도                | Archive 참조 차단 |
| 5   | Bug 이슈와 Enhancement 이슈를 동시에 같은 Feature에 등록        | 라벨 상호 배타성  |
| 6   | PRD 작성 중 사용자가 중간에 대화를 중단                         | 부분 생성 처리    |
| 7   | screens.yml 수정 후 IA 자동 집계 트리거                         | 자동화 정합성     |
| 8   | 2개 Feature가 동일한 화면 ID를 참조                             | 화면 충돌 감지    |
| 9   | BMAD-METHOD 워크플로우 실행 중 에러 발생                        | 에러 복구         |
| 10  | 기존에 수동으로 만든 GitHub Issue를 Feature 계층에 편입         | 마이그레이션      |
| 11+ | (추가 시나리오를 직접 도출해라)                                 |                   |

### B. 각 시나리오별 분석 형식

```
### 시나리오 #N: {제목}

**입력 조건**: (어떤 상태에서 시작하는지)
**실행 행동**: (사용자가 무엇을 하는지)
**기대 결과**: (워크플로우가 어떻게 반응해야 하는지)
**현재 설계의 처리 여부**: ✅ 처리됨 / ⚠️ 부분 처리 / ❌ 미처리
**미처리 시 개선 방안**: (어떻게 보완하면 되는지)
```

### C. 종합 개선 보고

시나리오 테스트 결과를 종합하여 **카테고리별로 그룹핑한 개선 제안 보고서**를 작성해라:

1. **즉시 반영 필요** (워크플로우가 깨지는 치명적 이슈)
2. **계획서에 추가 필요** (Phase에 태스크로 추가해야 할 항목)
3. **향후 개선 권장** (v1 이후 개선 가능한 항목)

## STEP 7: 불확실한 지점 질문

계획을 세우다가 **기획 문서에 정의되지 않았거나 모호한 부분**이 있으면, 가정을 세우지 말고 반드시 나에게 질문해라. 특히:

- BMAD-METHOD에서 가져올 워크플로우의 **최종 선별 승인**
- 미정의 항목(PRD 템플릿 세부 양식, screens.yml 스키마 등)에 대해 내가 결정해야 하는 부분
- 기존 에이전트의 동작을 수정해야 하는 경우 **허용 범위**
- GitHub Project 필드 추가/변경이 필요한 경우 **승인 여부**
- CIS 에이전트(BMad)의 실제 구현체 위치와 호출 방식
- BMAD-METHOD 산출물 경로 중 **PM-DOCS 내 매핑이 불확실한 경로**

**질문은 STEP별로 묶어서 한 번에 제시**하고, 내 답변을 받은 후 계획을 확정해라.

## 제약 조건

1. **코드를 작성하지 마라.** 계획서만 작성한다.
2. **기존 시스템을 먼저 분석**하고, 그 위에 새 워크플로우를 얹는 방향으로 설계한다. 기존 동작을 깨뜨리는 변경은 최소화한다.
3. **기존 워크플로우 변경 시 반드시 변경 사유를 명시**해라 — 대상 파일, 변경 전/후 동작, 사유, 영향 범위, 롤백 방안.
4. **PM-DOCS/ 폴더는 .claude/context/와 완전히 분리**된 독립 경로여야 한다. CI 자동 생성 경로와 절대 겹치지 않아야 한다.
5. **BMAD-METHOD의 산출물 경로는 PM-DOCS/ 내로 오버라이드**해야 한다. 원본 경로를 그대로 사용하지 마라.
6. 계획의 각 Phase는 **독립적으로 배포 가능**해야 한다. Phase 1만 완료해도 기존 시스템이 정상 작동해야 한다.
7. `Definition_of_Plan-based-workflow.md`의 §10(핵심 제약 조건) 9개 항목을 **불변 규칙**으로 취급하고, 계획이 이를 위반하지 않는지 최종 검증해라.
8. **미정의 항목은 STEP 5에서 철저하게 찾아내되**, 가정으로 채우지 말고 나에게 질문해라.
