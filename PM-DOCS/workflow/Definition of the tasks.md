# 프로젝트 작업 계층 구조 정의서

> **최종 수정일**: 2025-02-17
> **작성 목적**: Milestone → Feature → GitHub Issue 3단계 계층 구조와 상태 전파 규칙을 정의하여, 워크플로우 자동화 설계의 기준 문서로 활용한다.

---

## TL;DR

프로젝트의 작업 계층 구조는 **Milestone → Feature → GitHub Issue** 3단계로 구성된다. Milestone은 기간 목표, Feature는 에픽 단위 기능, GitHub Issue는 개발자가 가져가는 원자 작업이다. 각 계층은 상위 속성을 반드시 가져야 하며, 상태(backlog/ongoing/done) 전파 규칙이 계층별로 다르다.

- **Milestone**: 수동 변경 (의도적 수동 유지 확정)
- **Feature**: 하위 Issue 상태 기반 자동 전파
- **GitHub Issue**: 개발자 수동

---

## 1. Milestone (마일스톤)

| 항목               | 정의                                                                              |
| ------------------ | --------------------------------------------------------------------------------- |
| **정의**           | 서비스의 **기간 내 달성해야 할 목표** 단위                                        |
| **상위 속성**      | 없음 (최상위 계층)                                                                |
| **하위 속성**      | Feature (1:N)                                                                     |
| **상태 값**        | `backlog` → `ongoing` → `done`                                                    |
| **상태 변경 방식** | **수동** — 사용자가 직접 변경 (의도적 수동 유지)                                  |
| **저장 위치**      | `PM-DOCS/planning/Roadmap/`                                                       |
| **파일 규칙**      | 마일스톤 1개 = 파일 1개                                                           |
| **파일 내용**      | 마일스톤 메타데이터 + 연결된 Feature 목록 + 각 Feature에 연결된 GitHub Issue 목록 |

### 수동 유지 사유

PM이 직접 목표 달성 여부를 판단해야 하는 성격의 항목이므로, 하위 Feature/Issue 상태와 무관하게 PM의 주관적 판단에 의해서만 상태가 변경된다.

---

## 2. Feature (피쳐)

| 항목               | 정의                                                    |
| ------------------ | ------------------------------------------------------- |
| **정의**           | 마일스톤 하위의 **서비스 에픽 기능 단위** (중규모 이상) |
| **예시**           | 로그인/회원가입, 홈화면, 에디터 기능 추가 등            |
| **상위 속성**      | Milestone (**필수** — 반드시 1개의 마일스톤에 소속)     |
| **하위 속성**      | GitHub Issue (1:N)                                      |
| **상태 값**        | `backlog` → `ongoing` → `done`                          |
| **상태 변경 방식** | **자동** — 하위 GitHub Issue 상태 기반 전파             |

### Feature 상태 자동 전파 규칙

| 조건                          | Feature 상태 |
| ----------------------------- | ------------ |
| 배정된 GitHub Issue가 **0개** | `backlog`    |
| Issue **1개라도** 진행중      | `ongoing`    |
| 배정된 **모든 Issue**가 done  | `done`       |

### Feature 생성 시 부수 효과

- Feature 생성 → **GitHub Issue 라벨**로 동일 이름 자동 생성

---

## 3. GitHub Issue (깃헙 이슈)

| 항목               | 정의                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| **정의**           | 개발자가 가져가는 **원자 단위 일감**                                             |
| **상위 속성**      | Feature (**필수**, 라벨로 표현) + Milestone (**필수**, GitHub 마일스톤으로 표현) |
| **하위 속성**      | Sub-Issue (자동 생성 가능)                                                       |
| **생성 방식**      | Feature의 PRD를 TaskManagement 시스템이 분해 → Issue + Sub-Issue **자동 생성**   |
| **상태 변경 방식** | 개발자 수동 또는 GitHub 기본 워크플로우                                          |

### Issue 생성 시 자동 배정 규칙

| 속성         | 자동 배정 소스             |
| ------------ | -------------------------- |
| **라벨**     | Feature 이름               |
| **마일스톤** | Feature가 소속된 Milestone |

---

## 상태 전파 흐름도

```
Milestone (수동 변경)
 └─ Feature (자동: 하위 Issue 상태 기반)
     └─ GitHub Issue (개발자 수동)
         └─ Sub-Issue (개발자 수동)

상태 전파 방향: ↑ (아래에서 위로 자동 집계)
─────────────────────────────────────────────
Issue 전부 미배정        → Feature = backlog
Issue 1개라도 in progress → Feature = ongoing
Issue 전부 done          → Feature = done
Milestone                → PM 수동 판단 후 변경
```

---

## 속성 연결 자동화 요약

| 트리거                | 자동 수행 액션                                                                                     |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| Feature 생성          | ① GitHub 라벨 생성 (Feature명)                                                                     |
| Issue 생성 (PRD 분해) | ① 라벨 자동 배정 (Feature명) ② 마일스톤 자동 배정 (Feature의 상위 Milestone) ③ Sub-Issue 자동 생성 |
| Issue 상태 변경       | ① Feature 상태 자동 재계산 (`backlog` / `ongoing` / `done`)                                        |
| Milestone 상태 변경   | ① **수동만 허용** (자동 전파 없음)                                                                 |

---

## 핵심 제약 조건

1. **Feature는 반드시 Milestone에 소속**되어야 한다. Milestone이 없는 Feature는 생성할 수 없다.
2. **GitHub Issue는 반드시 Feature(라벨)와 Milestone을 가져야** 한다. 둘 중 하나라도 없으면 유효하지 않은 이슈로 간주한다.
3. **Milestone 상태는 자동 전파하지 않는다.** PM의 의도적 수동 관리 영역이다.
4. **Feature 상태는 하위 Issue 상태에 의해 자동 결정**되며, 수동 오버라이드는 허용하지 않는다.
